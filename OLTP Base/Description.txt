1. Загрузка данных из CSV в STAGING слой
Для загрузки исходных данных в OLTP‑базу используется промежуточный слой staging.
На этом уровне создаются временные таблицы, структура которых полностью соответствует структуре CSV‑файлов.

Данные из CSV загружаются в staging‑таблицы с помощью стандартного механизма pgAdmin Import .
На этом этапе данные не валидируются и не очищаются — staging служит только для временного хранения «сырых» данных.

Используемые staging‑таблицы:

staging_users

staging_teams

staging_team_members

staging_sports

staging_locations

staging_event_statuses

staging_events

staging_event_registrations

staging_payments

staging_reviews

Каждая таблица содержит только те поля, которые присутствуют в CSV.

2. ETL: очистка и подготовка данных в STAGING
После загрузки CSV выполняется серия шагов по очистке данных:

Удаление дубликатов  
Используется сравнение строк по ключевым полям и ctid, чтобы оставить только одну запись.

Удаление пустых и некорректных строк  
Удаляются строки с NULL‑значениями в обязательных полях.

Проверка внешних ключей  
Например:

капитан команды должен существовать в Users

TeamID должен существовать в Teams

EventID должен существовать в Events

RegistrationID должен существовать в EventRegistrations

Логическая валидация  
Например:

JoinDate > CreatedDate команды

Rating между 1 и 5

FK‑значения должны существовать в соответствующих таблицах

После очистки staging содержит только корректные и готовые к загрузке данные.

3. ETL: загрузка данных из STAGING в OLTP (UPSERT)
Загрузка данных в OLTP выполняется с помощью rerunnable UPSERT‑логики:

новые записи вставляются

существующие обновляются только если данные изменились

неизменённые записи не трогаются

Это достигается с помощью:

ON CONFLICT DO UPDATE

сравнения старых и новых значений в WHERE

LEFT JOIN + WHERE t.id IS NULL для таблиц без уникальных ключей

Примеры:

Users — конфликт по Email

Teams — конфликт по TeamName

TeamMembers — конфликт по (TeamID, UserID)

Events — уникальность по набору FK + EventDate

Payments — уникальность по RegistrationID + PaymentDate

Таким образом, скрипт можно запускать многократно — результат всегда будет корректным.

4. Итоговый процесс загрузки
Пользователь загружает CSV‑файлы в staging‑таблицы.

Выполняется очистка staging:

удаление дублей

удаление пустых строк

проверка FK

логическая валидация

Выполняется ETL‑скрипт, который:

вставляет новые данные

обновляет изменённые

пропускает неизменённые

OLTP‑база получает полностью корректные, чистые и согласованные данные.

5. Соответствие требованиям задания
Скрипт rerunnable

Данные не перезаписываются, если не изменились

Используется staging‑слой

Реализована полная ETL‑цепочка

Все таблицы OLTP корректно загружаются
